# TODO for Agent

Нужно безопасно реорганизовать структуру проекта без изменения поведения сайта.

## Цель

- улучшить читаемость и масштабируемость проекта;
- сократить размеры перегруженных файлов;
- вынести бизнес-логику из UI и `route.ts`;
- не сломать текущий UI, API и интеграции;
- сохранить все текущие фичи и визуальное поведение.

## Ограничения

- сначала изучи проект и существующие связи, не делай массовых переносов вслепую;
- не меняй поведение интерфейса, тексты, стили и API-контракты без необходимости;
- не удаляй пользовательские изменения, если они не относятся к задаче;
- делай рефакторинг поэтапно с промежуточной проверкой;
- после изменений обнови memory bank:
  - `.kilocode/rules/memory-bank/context.md`
  - при необходимости другие memory bank файлы.

## Порядок работы

### 1. Подготовка и аудит

- Просмотри текущее дерево проекта.
- Определи все точки входа и зависимости для:
  - `src/components/StreamSchedule.tsx`
  - `src/components/TwitchVods.tsx`
  - `src/app/api/twitch/route.ts`
  - `src/app/api/streams/route.ts`
  - `src/app/api/watch-also/route.ts`
  - `src/lib/*`
- Зафиксируй package manager, который реально используется, и наличие конфликтующих lock-файлов.
- Перед началом правок кратко сформулируй план по шагам.

### 2. Нормализация корня проекта

- Приведи `package.json` к реальному имени проекта вместо шаблонного.
- Выбери один package manager.
- Удали лишний lock-файл только если уверен, что он не нужен проекту.
- Добавь или обнови `README.md`:
  - что это за проект;
  - как запустить;
  - какие env-переменные нужны;
  - как устроены основные папки.
- Проверь `.gitignore`, чтобы локальные/IDE/build-артефакты были корректно исключены.

### 3. Введение feature-структуры без поломки импортов

- Создай целевую структуру:
  - `src/features/schedule/`
  - `src/features/twitch/`
  - `src/features/watch-also/`
  - `src/shared/`
  - `src/server/`
- Не переноси всё сразу. Сначала выделяй модули, потом переключай импорты.

### 4. Рефакторинг `StreamSchedule`

- Раздели `src/components/StreamSchedule.tsx` на меньшие части.
- Вынеси из него:
  - типы;
  - date/time утилиты;
  - построение календаря/недель;
  - клиентскую загрузку/синхронизацию данных;
  - desktop/mobile представления.
- Предпочтительная структура:
  - `src/features/schedule/components/StreamSchedule.tsx`
  - `src/features/schedule/components/ScheduleDesktop.tsx`
  - `src/features/schedule/components/ScheduleMobile.tsx`
  - `src/features/schedule/components/WeekPicker.tsx`
  - `src/features/schedule/lib/date.ts`
  - `src/features/schedule/lib/build-weeks.ts`
  - `src/features/schedule/lib/api.ts`
  - `src/features/schedule/types.ts`
- Сохрани текущий внешний API компонента, чтобы страница не сломалась.
- После переноса обнови импорт в `src/app/page.tsx`.

### 5. Рефакторинг `TwitchVods`

- Раздели `src/components/TwitchVods.tsx` на небольшие части.
- Вынеси:
  - типы;
  - форматтеры;
  - клиентский fetch/cache слой;
  - VOD shelf;
  - Clips shelf.
- Предпочтительная структура:
  - `src/features/twitch/components/TwitchVods.tsx`
  - `src/features/twitch/components/VodsShelf.tsx`
  - `src/features/twitch/components/ClipsShelf.tsx`
  - `src/features/twitch/lib/format.ts`
  - `src/features/twitch/lib/api.ts`
  - `src/features/twitch/types.ts`
- Сохрани текущее визуальное поведение и навигацию по страницам/кнопкам.

### 6. Серверный слой

- Вынеси бизнес-логику из API-роутов в `src/server/`.
- Целевая структура:
  - `src/server/db/`
  - `src/server/streams/`
  - `src/server/twitch/`
  - `src/server/kick/`
- Для `streams`:
  - SQL и доступ к БД вынеси в `repository.ts`;
  - нормализацию/валидацию и orchestration вынеси в `service.ts`.
- Для `twitch`:
  - получение токена, запросы к Twitch, чтение/синк кэша, сбор ответа вынеси из `route.ts`;
  - `route.ts` должен стать тонким адаптером.
- Для `watch-also`:
  - логику Twitch/Kick-агрегации вынеси в сервисы.
- Не меняй JSON-формат ответов API без необходимости.

### 7. Shared-слой

- Перенеси общие клиентские утилиты в `src/shared/lib/`.
- Перенеси переиспользуемые типы в `src/shared/types/`, если это действительно общее.
- Раздели клиентские и серверные модули так, чтобы серверный код не утекал в client bundle.
- Проверь, что алиасы `@/*` продолжают работать без дополнительных костылей.

### 8. Legacy и чистка структуры

- Если после переноса в `src/components/` остаются устаревшие или неиспользуемые файлы, не удаляй их агрессивно.
- Сначала проверь реальные импорты.
- Если файл не используется, либо:
  - удали его, если уверен;
  - либо перенеси в `src/legacy/` и явно отметь это.
- Не трогай медиа-ассеты и API-маршруты, которые не относятся к рефакторингу.

### 9. Проверка после каждого этапа

- После каждого крупного этапа проверяй:
  - `npm run lint`
  - `npm run typecheck`
  - при наличии `npm run build`
- Если в проекте используется не `npm`, применяй фактический package manager.
- Не считай задачу завершённой, пока проект не проходит хотя бы lint + typecheck.
- Если build падает из-за уже существующих проблем, отдели свои изменения от исходных проблем и явно опиши это.

### 10. Финализация

- Обнови imports и удаляй временные переходные файлы только после успешной проверки.
- Обнови memory bank.
- В финальном отчёте дай:
  - что именно было реорганизовано;
  - какие файлы и папки появились;
  - что осталось без изменений;
  - результаты проверок;
  - возможные остаточные риски.

## Критерии приёмки

- приложение визуально и функционально работает как раньше;
- `src/components/StreamSchedule.tsx` больше не монолит;
- `src/components/TwitchVods.tsx` больше не монолит;
- `src/app/api/twitch/route.ts` и `src/app/api/streams/route.ts` стали тонкими;
- серверная логика вынесена в `src/server/`;
- общие утилиты вынесены в `src/shared/`;
- структура проекта стала понятнее без изменения пользовательского поведения;
- memory bank обновлён.

## Важно

- если в процессе обнаружишь неожиданные пользовательские изменения или конфликтующие правки, остановись и опиши риск до продолжения;
- если придётся выбирать между "красивым" рефакторингом и безопасным рефакторингом, выбирай безопасный.
