# TODO for Agent

Нужно добавить отдельную страницу для анонимной оценки стримов, где любой человек может оценить конкретный стрим один раз без регистрации и без сбора персональных данных. Главную страницу и блок расписания не нужно перегружать UI голосования.

## Цель

- вынести оценку стримов на отдельную страницу, чтобы не захламлять контент главной;
- дать пользователям возможность перейти к рейтингу из понятной точки входа;
- обеспечить ограничение "один голос на один стрим" на уровне браузера/устройства;
- не собирать имя, email, телефон, аккаунт, fingerprint или IP в базе;
- встроить решение в существующие `streams` и серверный слой без ломки текущего расписания.

## Принцип ограничения без персональных данных

- идентификация строится не на IP и не на аккаунте, а на случайном анонимном токене в `HttpOnly` cookie;
- сервер хранит в базе только хеш токена, а не сам токен;
- один и тот же cookie может поставить оценку конкретному стриму только один раз;
- смена IP не должна влиять на возможность голосования;
- очистка cookie, другой браузер или другое устройство будут считаться новым анонимным пользователем.

## Ограничения

- не использовать авторизацию и не добавлять обязательный аккаунт;
- не хранить raw IP, email, user-agent fingerprint или любые иные PII;
- не встраивать полный UI голосования в карточки расписания на главной странице;
- главная страница может содержать только легкую точку входа, если она вообще понадобится;
- все изменения делать поэтапно с проверкой типов и линта;
- после реализации обновить memory bank:
  - `.kilocode/rules/memory-bank/context.md`
  - при необходимости `.kilocode/rules/memory-bank/tech.md`
  - при необходимости другие memory bank файлы, если изменится архитектурное описание.

### 3. Модель данных для рейтингов

- Минимальный состав полей:
  - `id`
  - `stream_id`

- Добавь ограничения:

### 4. Серверная анонимная идентификация

- Реализуй helper для чтения/создания анонимного токена голосующего.
- Токен должен:
  - быть случайным криптографически стойким значением;
  - храниться в `HttpOnly` cookie;
  - иметь долгий срок жизни;
  - не содержать персональных данных.
- Перед записью в БД хешируй токен на сервере, например через `sha256`.
- Не логируй raw token в консоль и не возвращай его клиенту.

### 5. Репозиторий и сервис рейтингов

- Расшири серверный слой для работы с рейтингами:
  - запрос списка стримов с агрегатами рейтинга;
  - получение текущей оценки пользователя для конкретного стрима;
  - запись новой оценки;
  - при выбранной политике опционально обновление своей предыдущей оценки.
- Явно выбери одну из политик и придерживайся ее во всем коде:
  - либо "голос только один раз без изменения";
  - либо "голос один раз, но свою оценку можно заменить".
- Если политика будет второй, `UNIQUE` остается, а запись идет через upsert.

- Не завязывай ограничение на IP.

- Если будет rate limit, убедись, что он не ломает обычный сценарий голосования.
- На странице оценки явно зафиксируй фактическое ограничение:
  - это один голос на браузер/устройство, а не гарантированно один голос на реального человека.

## Предлагаемая схема БД

```sql
CREATE TABLE IF NOT EXISTS stream_ratings (
  id BIGSERIAL PRIMARY KEY,
  stream_id BIGINT NOT NULL REFERENCES streams(id) ON DELETE CASCADE,
  viewer_token_hash TEXT NOT NULL,
  rating SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (stream_id, viewer_token_hash)
);

CREATE INDEX IF NOT EXISTS stream_ratings_stream_id_idx
ON stream_ratings (stream_id);
```

## Критерии приёмки

- пользователь может открыть отдельную страницу оценки стримов;
- пользователь может оценить реальный стрим вне главной страницы;
- один и тот же анонимный cookie не может бесконечно голосовать за один и тот же стрим;
- смена IP не создает новый голос сама по себе;
- в базе не хранятся имя, email, raw IP, аккаунт или browser fingerprint;
- сервер хранит только хеш анонимного токена;
- главная страница не захламлена интерактивным рейтингом;
- страница оценки показывает среднюю оценку, число голосов и состояние текущего пользователя;
- `npm run lint` и `npm run typecheck` проходят;
- memory bank обновлен.

## Важно

- если в процессе окажется, что нужна более строгая модель "один человек = один голос", это уже требует auth или внешнего trust-механизма;
- если придется выбирать между полной строгостью антинакрутки и отсутствием персональных данных, приоритет у приватности и прозрачного ограничения модели;
- если обнаружатся неожиданные пользовательские изменения в расписании или API, сначала опиши риск, потом продолжай.
